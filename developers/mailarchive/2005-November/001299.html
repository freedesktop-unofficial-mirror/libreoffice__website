Hi Martin,

	So - as you may know we have a 'src-pack' shell script:

	http://go-oo.org/ooo-build/bin/src-pack

	It splits the OO.o source checkout into several pieces, with the net
effect that developers typically only need to download something half
the size.

	With my latest set of tweaks we have:

	-core:     ~114m
	-lang:      ~52m
	-sdk_oo:    ~29m
	-system:    ~22m
	-binfilter: ~18m

	Of course - ideally, in future, we could perhaps version such a split
setup, and only release new -binfilter, -system, etc. packages when
things actually changed there across milestones; but - for now, this is
perhaps a good compromise.

	So - my question is: can we host this on an 'approved' OO.o site, or
alternatively - bless the go-oo.org milestone snapshots as something
'official' ? [ perhaps with a DNS re-direct ? ].

	Does it make sense to split up our official source archives in this way
as an optional extra ?

	Rene - since you're a major driver of the -system patch sets etc. it'd
be great to poke at http://go-oo.org/ooo-build/doc/code-sizes.txt which
shows the remaining worst-offenders.

	boost, icu, berkeleydb, xalan, stlport

	We should prolly work on trying to migrate them to the -system package,
as/when systems are prevalant with the right versions installed on them.
[ it seems not even Debian is using --system-icu or --system-boost ].

	Martin - what do you think ?

	Regards,

		Michael.
