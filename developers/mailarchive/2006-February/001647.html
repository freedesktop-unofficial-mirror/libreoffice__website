I've finally had a bit of time to look into this -- the NullPointerException 
happens in xt*s com.jclark.xsl.sax.XMLProcessorImpl.load()'s

parser.setDocumentHandler(builder);

Because at that time, parser is null.

The parser is set before in 
xmlhelp/source/com/sun/star/help/HelpIndexer.java's ParseStuff constructor 
without checking for a null value -- so the obvious fix is:

--- xmlhelp/source/com/sun/star/help/HelpIndexer.java.ark	2006-02-26 
21:10:20.000000000 +0100
+++ xmlhelp/source/com/sun/star/help/HelpIndexer.java	2006-02-26 
21:11:22.000000000 +0100
@@ -537,7 +537,10 @@
 			SAXParserFactory spf = SAXParserFactory.newInstance();
 			spf.setValidating(false);
 			try {
-				_processor.setParser(spf.newSAXParser().getParser());
+				Parser p=spf.newSAXParser().getParser();
+				if(p == null)
+					throw new java.lang.Exception("newSAXParser returned a null parser");
+				_processor.setParser(p);
 			} catch (java.lang.Exception e) {
 				System.err.println(
 					"<!-- NO HELP AVAILABLE: no parser found -->");

Unfortunately that just hides the bigger problem (why the **** is 
newSAXParser() returning something bogus?) -- we should probably add this (or 
something similar) anyway because the NullPointerException inside Xt is not 
exactly a meaningful error message.


Digging a bit deeper, the real problem seems to be in classpath -- unless 
explicitly overridden in a property, classpath's 
javax.xml.parsers.SAXParserFactory.newInstance() returns a 
gnu.xml.stream.SAXParserFactory; the SAXParser generated by it is a 
gnu.xml.stream.SAXParser.

gnu.xml.stream.SAXParser's idea of implementing getParser() is

public Parser getParser()
  throws SAXException
{
  return null;
}

Somehow this can't be quite right...

I guess the right workaround for now is to force HelpLinker to use an 
org.apache.xerces.jaxp.SAXParserFactoryImpl as opposed to a 
gnu.xml.stream.SAXParserFactory -- any objections to this?
