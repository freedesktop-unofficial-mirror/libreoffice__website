Hi Anil,

	The alpha issue is - that we have to have the alpha separate from the
BitmapColor that is used to transfer alpha around the place - since
Windows relies on that being the size it is now for various nasty
memcopy operations - so we can't change it.

	In the past people have assumed they could copy an 8bit - paletized
image around containing a magic color that can be  mapped to a 1 bit
alpha mask right before render. With a separate 8bit alpha channel this
assumption gets badly broken :-) Thus we have to retro-fit all code with
that assumption to use a BitmapEx instead of a Bitmap - since this
contains the Alpha mask, and to construct that nicely from the resource
otherwise we loose alpha, the default alpha is 0 - and we end up with a
beautiful icon, at 0 alpha == invisible ;-) [ or something worse if the
icon happened to be paletized beforehand - a pink outlined mess eg. ;-]

	Anyhow, hope that helps.
